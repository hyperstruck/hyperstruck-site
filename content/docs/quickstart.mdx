This is the fastest first-success path: upload a provider credential, create an agent, dispatch a goal, and optionally store a manual learning.

<Callout type="warning" title="An API key alone is not enough">
  A fresh Hyperstruck API key does not let an agent run by itself. The agent also needs a matching active provider credential for its `model_provider`. For most teams, a `tenant_default` credential is the fastest setup path. See [Credentials API](/docs/credentials-api).
</Callout>

<Callout type="info" title="Keep the first call small">
  Start with the minimum required fields. Full API reference lives in the OpenAPI docs. If you omit optional fields here, Hyperstruck uses sane defaults.
</Callout>

## Minimal flow

<Steps>
  <Step>

### 1. Upload a tenant-default provider credential

Use a tenant default first so any agent with the same `model_provider` can run without extra credential setup.

```bash
curl -X POST \
  -H "Authorization: Bearer <API_KEY>" \
  -H "Content-Type: application/json" \
  "https://api.core.hyperstruck.com/credentials/providers" \
  -d '{
    "provider": "anthropic",
    "display_name": "Default Anthropic Key",
    "binding_type": "tenant_default",
    "secret": "<ANTHROPIC_API_KEY>"
  }'
```

  </Step>
  <Step>

### 2. Create an agent

Create an agent with the required fields plus an optional `description`. `description` is a short string describing the agent's purpose or area of expertise in more detail.

```bash
curl -X POST \
  -H "Authorization: Bearer <API_KEY>" \
  -H "Content-Type: application/json" \
  "https://api.core.hyperstruck.com/agents" \
  -d '{
    "name": "Sales Execution Assistant",
    "description": "Helps sales reps plan follow-ups, next steps, and account strategy.",
    "model_provider": "anthropic",
    "model_name": "claude-sonnet-4-5",
    "core_config": {
      "instructions": "You are a helpful sales execution assistant."
    }
  }'
```

  </Step>
  <Step>

### 3. Dispatch a goal

Use the returned `AGENT_ID` to start a run.

```bash
curl -X POST \
  -H "Authorization: Bearer <API_KEY>" \
  -H "Content-Type: application/json" \
  "https://api.core.hyperstruck.com/agents/<AGENT_ID>/goals" \
  -d '{
    "goal": "Create a follow-up sales execution plan for a call with client XYZ"
  }'
```

  </Step>
  <Step>

### 4. Optionally store a manual learning

Reasoning runs can already extract learnings automatically. Use manual storage when you want direct editorial control.

```bash
curl -X POST \
  -H "Authorization: Bearer <API_KEY>" \
  -H "Content-Type: application/json" \
  "https://api.core.hyperstruck.com/agents/<AGENT_ID>/learnings" \
  -d '{
    "content": "At client XYZ, directors and below are usually influencers, not final decision-makers, so map the buying committee before assigning next steps.",
    "learning_type": "pitfall"
  }'
```

  </Step>
  <Step>

### 5. Reuse the same capabilities from developer tooling

Once the API flow works, you can expose the same capabilities in developer tooling.

```text
/hyper-reasoning Create a follow-up sales execution plan for a call with client XYZ
/hyper-learning search client XYZ decision makers
/hyper-learning store
```

  </Step>
</Steps>

## Common next knobs

- Agent creation defaults: `status="active"`, `reasoning_profile="balanced"`, `memory_profile="default"`, `knowledge_scope="agent"`
- Goal dispatch optional fields: `context`, `session_id`, `worker_profile`, `metadata`
- Manual learning optional fields: `confidence`, `source_goal`, `applicable_goals`, `applicable_tools`, `privacy`

## Why credentials come first

Credential resolution is provider-based:

1. Hyperstruck looks at the agent's `model_provider`.
2. A matching active `agent_override` credential for that agent wins first.
3. If none exists, Hyperstruck falls back to the matching active `tenant_default`.
4. If neither exists, agent runs cannot start and manual learning store requests for that agent fail.

Use [Credentials API](/docs/credentials-api) for the full credential setup and resolution rules.
